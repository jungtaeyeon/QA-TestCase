# 1 Policy

## **1.1** **계정 ID**

* 계정 ID는 이메일 주소를 사용하며, 최초 생성 이후 변경이 불가하다.
* 계정 생성 시 시스템 전체 범위에서 중복 여부를 검증한다.

## **1.2 초대 기반 계정 등록**

계정은 관리자의 초대를 통해서만 생성되며, 자가 가입(Self-signup)은 지원하지 않는다.

* 최초 Master Admin: Backoffice Admin이 관련 정보 입력하고 Master Admin 대상자를 초대
* 이후 계정 생성/초대 가능 주체: Master Admin, Org Admin. (Org user(App Admin, App user) 이하는 초대 불가)

  + Org Admin은 Master Admin이 기존 계정에 Org Admin 역할을 부여하여 생성됨.
* 초대 재발송 가능 주체: Master Admin / Org Admin만 가능.

## **1.3. 계정 상태 정의**

계정 상태는 4가지로 구분되며, 조건에 따라 자동 또는 수동으로 전환된다.

|  |  |
| --- | --- |
| 상태 | 정의 및 전환 조건 |
| Invitation | 초대 메일 발송 후 비밀번호 미설정 상태. 관리자 초대 시 자동 부여. |
| Active | 정상 사용 가능 상태. 비밀번호 최초 설정 완료 시 자동 전환. |
| Inactive | 관리자에 의해 접근이 차단된 상태. 상위 관리자가 수동 변경. |
| Locked | 로그인/MFA 실패 누적으로 시스템이 자동 차단. 5회 연속 실패 시 자동 전환. |

* Locked 상태는 Inactive와 독립적으로 관리되며, 잠금 해제 후 Active로 복귀한다.
* Master Admin 계정은 Inactive 전환이 불가하며, 계약 종료 시 Backoffice에서만 처리 가능하다.

# 2 Account Model

## **2.1 초대 흐름**

1. 관리자가 초대 대상 이메일 입력 및 Org 역할 지정
2. 시스템이 초대 메일 발송 (계정 상태: Invitation)
3. 초대받은 사용자가 메일 내 링크 클릭 후 최초 정보 및 비밀번호 설정
4. 설정 완료 즉시 계정 상태 Active 전환, 초대 링크 즉시 파기

* 초대 링크 유효기간: 발송 후 7일. 기간 내 미수락 시 자동 만료.
* 재발송 시 이전 링크 자동 파기, 최신 링크만 유효.

## **2.2 App 접근 권한 결정 구조**

사용자가 특정 App에 최종적으로 접근하려면 아래 3단계 조건이 모두 충족되어야 한다. 하나라도 충족되지 않으면 접근 불가.

|  |  |  |  |
| --- | --- | --- | --- |
| 단계 | 조건 | 관리 주체 | 관련 섹션 |
| 1단계 | 계약에 App 포함 | Backoffice Admin (계약 생성 시 설정) | 계약 관리 (Backoffice) |
| 2단계 | Workspace에서 App 활성화 | Backoffice Admin | Backoffice 2.3 |
| 3단계 | 사용자에게 App 접근 권한 부여 | Backoffice Admin , Master Admin , Org Admin(App Admin은 부여 불가) | 9.2 계정 생성 / 9.5 계정 정보 수정 / UM40, UM45 |

*※ 3단계 중 하나라도 미충족 시 해당 App 접근 불가. 2단계(Workspace 활성화) 여부와 무관하게 3단계 권한이 부여되어 있더라도 2단계가 비활성이면 접근 불가.*

*※ 앱 접근 권한 부여(3단계)는 Backoffice Admin / Master Admin /* Org *Admin만 가능. App Admin은 부여 불가.*

## **2.3 사용자-Workspace 다중 접근**

* 사용자는 주어진 접근 권한에 따라 1개 혹은 최대 N개 워크스페이스에 접근할 수 있다.
* 사용자-Org 다중 접근: 소속 Org가 2개 이상인 경우 Org 전환 시 기존 세션 종료 및 신규 세션 발급.

## 2.4 조회 권한 범위 (페이지/기능 기준)

조회 권한은 사용자에게 부여된 Workspace 및 App 권한을 기준으로 제한된다.

Org 정보 조회

* 관리자 역할(Master Admin, Org Admin)은 Org Admin에 접근하여 Org 관련 정보를 조회할 수 있다.
* 일반 사용자(Org User)는 Org Admin 접근이 제한된다.

Workspace 목록 및 정보 조회

* 사용자는 본인이 접근 권한을 가진 Workspace에 한해서만 조회할 수 있다.
* Workspace 목록의 노출 방식(전체 표시 또는 권한 기반 필터링)은 네비게이션 정책에 따른다.

App 목록 및 접근

* 사용자는 Workspace 내에서 권한이 부여된 App만 조회 및 접근할 수 있다.
* 권한이 없는 App은 조회 또는 접근이 제한된다.
* App 노출 방식은 네비게이션 정책에 따른다.

사용자 정보 조회

* 관리자 역할은 Org 내 Admin에 접근하여 사용자 목록 및 상세 정보를 조회할 수 있다.
* 일반 사용자(Org User)는 Org Admin 접근이 제한된다.

App 데이터 조회

* 사용자는 접근 권한이 부여된 App의 데이터(Safety DB 등)만 조회할 수 있다.
* 조회 범위는 Workspace 및 App 권한에 의해 제한된다.

App Admin 권한

* App Admin은 담당 Workspace의 특정 App에 대한 설정 및 관련 정보를 조회할 수 있다.
* App Admin은 사용자 접근 권한 자체는 조회 및 변경할 수 없다.

# 3 Authentication & Security Policy

## **3.1 Invitation 상태에서의 인증 실패 처리**

|  |
| --- |
| 정책: 초대받은 사용자가 비밀번호 최초 설정 화면에서 5회 연속 실패한 경우,  계정은 Locked 전환 없이 Invitation 상태를 유지하며 초대 링크만 무효화된다.  관리자가 초대를 재발송해야 하며, 이전 링크는 자동 파기된다.  Locked 처리는 Active 상태 이후 로그인 시도에서만 적용된다. |

## **3.2. 비밀번호 정책**

### **3.2.1 복잡도 요건**

* 길이: 최소 9자, 최대 20자
* 대문자, 소문자, 숫자, 특수문자 각 1개 이상 필수 포함
* 4자 이상 연속된 동일 문자열 사용 금지 (예: aaaa, 1234, abcd)
* 직전 사용 비밀번호 재사용 금지

### **3.2.2 변경 주기 (변경 주기는 고라이브 시점에 맞춰서 개발한다: 참고** [https://selta.atlassian.net/wiki/spaces/IS/pages/23822337/Account+Login+Password#%EC%9D%BC%EC%A0%95-%EA%B5%AC%EB%B6%84](https://selta.atlassian.net/wiki/spaces/IS/pages/23822337/Account%2BLogin%2BPassword#%EC%9D%BC%EC%A0%95-%EA%B5%AC%EB%B6%84) )

* 최초 설정일로부터 180일마다 변경 요구. 181일째 로그인 시 변경 페이지 강제 표시.
* 변경 전까지 서비스 접근 제한하고. '나중에 변경' 유예 최대 5회 허용하되, 5회 초과 시 강제 변경.
* 모든 계정에 180일 주기가 예외 없이 적용된다.

### **3.2.3 재설정**

* 사용자: 로그인 페이지 또는 개인 프로필에서 재설정 요청 시 등록 이메일로 링크 발송
* 관리자: 관리자 화면에서 재설정 메일 발송만 가능. 직접 비밀번호 지정 불가.

## **3.3. MFA 정책**

### **3.3.1 설정 단위 및 주체**

* MFA는 Org 단위로 활성화/비활성화하며, 개인별 On/Off는 지원하지 않는다.
* 설정 가능 주체: Backoffice Admin, Master Admin, Org Admin.

### **3.3.2 인증 방식 및 흐름**

* 인증 수단: 이메일 기반 6자리 코드(숫자). 유효기간 10분.
* 코드 재발급: 30초 쿨다운 적용. 30초 내 재발급 시도 시 오류 팝업 표시.
* 코드 파기: 새 코드 발급 시 이전 코드 즉시 파기.
* 잠금 처리 5회 연속 실패 시 계정 자동 Locked 전환. 6번째 시도 시 잠금 안내 메시지 표시.
* 참고

  + 싱글 Org인 경우에는 로그인 후 MFA를 인증하나, 멀티 Org인 경우에는 로그인 후 조직을 고르고 나서 해당 조직이 MFA 활성화 되어 있는 경우에 인증 진행

### **3.3.3 MFA 설정 변경 시 기존 세션 처리**

* MFA 정책 변경은 **다음 로그인부터 적용**된다. 기존 활성 세션은 영향받지 않는다. 정책 변경 이벤트는 Audit Log에 기록된다.

## **3.4. 계정 잠금 및 해제**

* 비밀번호 입력 5회 연속 실패, 비밀번호 인증 성공 이후 MFA 5회 연속 실패 시 계정 자동 Locked 전환 (Invitation 상태 제외).
* 잠금 발생 시 해당 계정으로 로그인 화면에서 안내 메시지 표시.
* 사용자 본인 비밀번호 재설정 완료 시 잠금 자동 해제.

# 5 Session & Access Continuity

## **5.1 세션 유효시간**

* 로그인 후 세션 유효시간: 60분. 이후 자동 로그아웃.
* 초기화 조건: 특정 페이지 이동 또는 액션 발생 시 60:00으로 초기화

  + 대상 페이지: Safety DB 사이드바에 있는 메뉴 이동시, 보고 시작 시
* 만료 5분 전 팝업 표시: "아직 사용 중이신가요? 세션이 곧 만료됩니다. iVigilance Square를 계속 사용하려면 '계속 사용' 버튼을 클릭하세요." — 로그아웃 / 계속 사용 선택 가능.

  + 계속 사용 클릭 시 팝업 닫힘 + 60:00으로 초기화. 로그아웃 클릭 시 로그인 화면으로 이동.
* 5분 내 아무 응답 없을 시(00:00 도달) 자동 로그아웃 처리.
* 팝업 배경 클릭으로 닫히지 않음.

## 한도 계산 원칙

* Inactive / Invitation 상태 계정은 한도 카운트에서 제외한다.
* Viewer는 전체 Active, Locked 계정 수(200명) 카운트에 포함하지 않는다.

## **7.4 Admin User Management Flows**

### **7.4.1 계정 생성 흐름 (Admin > User Management)**

관리자가 User Management 화면에서 새 계정을 초대하는 흐름이다. 상세 정책은 4.2 계정 생성 참조. 특정 역할 한도가 도달한 경우 해당 역할이 드롭다운에서 선택 불가 상태로 표시된다. 전체 계정 수가 한도(200명)에 도달하면 [+ 계정 생성] 버튼 자체가 비활성화된다.

|  |
| --- |
| 📋 계정 생성 흐름 (Admin > User Management) |

### **7.4.2 역할 변경 흐름 (Admin > User Management > 계정 정보 수정)**

관리자가 기존 계정의 Org 역할 또는 App 역할을 변경하는 절차다. Org 역할 변경과 App 역할 변경은 별개 동작이다. 역할 변경은 즉시 적용되며, 하위 권한 Admin은 동등하거나 상위 권한 계정의 Org 역할을 변경할 수 없다.

역할 하향(강등) 시 처리 원칙:

* Org Admin → Org User 강등: Master Admin·Org Admin 전용 화면에 접속 중일 경우 권한 변경 팝업을 표시하고 Org 홈으로 이동한다.
* App Admin → App User 변경: 해당 계정이 App 어드민 화면에 접속 중일 경우 동일한 팝업을 표시하고 Org 홈으로 이동한다.

# 9 고도화 요구사항

본 섹션은 고도화 단계에 구현 예정인 요구사항들을 기록한다.

## 9.1 External User

External User는 고객사 외부의 관계인(CRO 등)이 Safety DB만 사용할 수 있도록 만든 역할이다. External User가 도입되면 한 사용자가 여러 Org에 소속되는 케이스 (예: 하나의 CRO 인원이 여러 제약사에 초대되는 경우)가 자연스럽게 발생하며, 이에 따라 계정 모델, 로그인 플로우, 세션, 화면 구성 등에 연쇄 변경이 필요하다. 아래 항목들은 External User 도입 시점에 함께 반영되어야 한다.

### 기존 개발에서 External User 추가 시 고려되어야 하는 부분

* 타 Org 소속 이메일이면 기존 계정에 권한 추가 플로우로 분기. 동일 Org 내 중복만 오류 처리(UM-11 이월)
* UM-12 역할 선택에 External User 포함. External User 50명 상한 초과 시 역할 선택 비활성화 (UM-12 이월)
* Active External User 50명 별도 상한은 전체 Active, Locked 200명 한도 내 포함 (UM-13 이월)
* LGN-44: External User 포함 전체 계정 180일 유예 동일 적용 (LGN-44 이월)
* 7.4.1 계정 생성 흐름: External User 최대 50명 한도 도달 시 선택 불가 처리 (7.4.1 이월)
* 역할 강등 케이스: App Admin → External User 강등, Org Admin → External User 강등 (7.4.2, ACC-12 이월)
* PWD-04: External User 포함 전체 계정 비밀번호 180일 정책 동일 적용 (PWD-04 이월)
* SES-07, SES-09 크로스 테넌트 관련 External User 동작은 본문 유지, 상태 "고도화"

### External User 추가에 따른 신규 기능 개발이 필요 한 부분

* 로그인 시 Org 선택 화면 (단일 Org면 스킵)

  + External User는 여러 Org에 동시에 소속될 수 있기 때문에, 로그인 시 어느 Org로 진입할지 결정해야 한다. 현재 로그인 플로우는 ID/PW 입력 후 바로 서비스 진입이지만, 멀티 Org 사용자는 ID/PW 입력 후 Org 선택 화면을 거쳐 진입하는 방식으로 변경되어야 한다. 단일 Org 사용자는 선택 화면을 건너뛰어 기존 플로우를 유지한다.
* MFA Org 설정 기반 적용

  + External User는 본인의 원래 소속 Org가 아닌 외부 Org로 접근하는 경우가 많기 때문에, MFA 적용 여부가 모호해진다. Org 단위로 MFA 설정을 관리하고, 사용자가 Org를 선택한 이후 해당 Org의 MFA 설정에 따라 인증 여부를 결정하는 방식으로 설계해야 한다.
* Org 전환 세션 관리

  + 한 사용자가 여러 Org에 접근할 수 있게 되면 Org 간 데이터 격리가 세션 레벨에서 보장되어야 한다. Org 전환 시 기존 세션을 종료하고 신규 세션을 발급하는 방식으로 처리한다. SES-05, SES-07~09와 연동
* Profile 소속 표시 방식

  + Profile 화면에는 현재 사용자의 소속 Org가 표시되는데, External User가 여러 Org에 속하는 경우 한 Org에서 볼 때 다른 Org 소속(경쟁사일 수 있음)이 노출되는 문제가 있다. Profile에서 Organization 항목을 제거하거나, 현재 접속 중인 Org 기준으로만 표시하는 방식으로 변경해야 한다. 현 릴리즈에서는 기존 기획을 유지한다.
* External User Role 구현 방식 (is\_external 플래그 등)

  + 기존 App User에 is\_external 플래그를 추가하는 방식으로 구현되어야 한다. 플래그 방식을 채택할 경우 Role 자체는 App User로 두고, 플래그 값에 따라 Safety DB 한정 접근, 50명 상한 등 제약을 적용한다. 초대 시점에 플래그를 설정하고 이후 변경은 불가능하도록 한다.
* Cross-Org 초대 시 기존 계정에 권한 추가 - 이메일 중복 처리: 타 Org 소속 이메일은 기존 계정에 권한 추가

  + External User가 여러 Org에 속하게 될 때, 각 Org마다 별도 계정을 생성하면 1인 다계정이 되어 GxP 감사추적 원칙에 맞지 않는다. 동일 이메일이 초대되는 경우 신규 계정을 생성하지 않고 기존 계정에 해당 Org 접근 권한을 추가하는 방식으로 처리한다.
* 이메일 중복 처리

  + 계정 생성 시 이메일이 시스템 전체에 이미 존재하는 경우, 기존에는 "이미 사용 중인 이메일입니다" 오류를 표시했다. Cross-Org 도입 후에는 타 Org 소속 이메일이면 기존 계정에 권한 추가 플로우로 분기하고, 동일 Org 내 중복인 경우에만 오류 처리한다. UM-11에서 이미 정책으로 기술되어 있으나 현 시점에서는 단일 Org 전제로 구현된다.

## 9.2 Viewer

Viewer는 감사(Audit) 목적으로 Org 전체의 로그를 조회만 할 수 있는 읽기 전용 역할이다. 감사 기간에 일시적으로 사용 후 감사 종료 시 비활성화 처리된다. 현재 릴리즈에서는 감사 시나리오 자체가 스코프 바깥이므로, Viewer 역할 구현은 고도화 단계로 이월한다.

### 기존 개발에서 Viewer 추가 시 고려되어야 하는 부분

* UM-12 역할 선택에 Viewer 포함. Viewer 5명 상한 초과 시 역할 선택 비활성화 (UM-12 이월)
* Viewer는 전체 Active, Locked 계정 수(200명) 카운트에 포함하지 않음 (UM-13, AP-01, ACC-14 이월)
* 7.4.1 계정 생성 흐름: Viewer 최대 5명 한도 도달 시 선택 불가 처리 (7.4.1 이월)
* SES-07, SES-09 크로스 테넌트 관련 Viewer 동작은 본문 유지, 상태 "고도화"

### Viewer 추가에 따른 신규 기능 개발이 필요 한 부분

* Viewer Role 구현

  + Viewer는 일반적인 Role과 달리 "읽기 전용으로 광범위한 범위를 조회 가능"한 특성이 있어, 권한 모델을 별도로 설계해야 한다. 기본적으로 Org 레벨 전체 조회를 허용하며, 특정 Workspace / App 단위 감사가 필요한 경우 Master Admin / Org Admin이 범위를 지정해 접근 권한을 추가 부여한다.
* Org 전체 로그 조회 권한 처리

  + Viewer는 Audit Trail, Activity Log, Audit Log 3종 모두를 Org 전체 범위에서 조회할 수 있다. 일반 Role은 App 단위 혹은 담당 범위에서만 조회 가능하므로, Viewer는 조회 권한 매트릭스에서 별도 경로로 처리된다. AUD 5.1 정책 표 참조.
* App 접근 시 App Admin / App User 권한 추가 부여

  + Viewer가 특정 App을 조회하려면 해당 App 접근 권한이 별도로 부여되어야 한다. 이때 App Admin 또는 App User 권한을 추가 부여하는 방식으로 구현한다. 단 Viewer 역할 특성상 부여된 App 권한으로도 쓰기·수정·삭제는 불가하며, 조회만 가능하도록 처리된다.
