# 개요 및 정책

* 목적: ICSR(개별 사례 안전성 보고서)의 식별 번호 생성 규칙을 표준화하여 데이터 정합성을 확보하고 관리 효율성을 증대함.
* 대상 필드: Case ID 필드. 추후 국가코드, 조직명을 포함하여 C.1.1 R3 필드에 매핑
* 필수로 설정해야 하는 룰

  + 설정 가능한 모든 텍스트는 영문 및 숫자만 가능하며 대소문자는 구분하여 입력 가능
  + ID 재사용 방지: 한 번 부여된 일련번호는 해당 케이스가 나중에 삭제되더라도 절대 재사용되지 않음

# Case ID의 개념

* 케이스 생성 시 자동으로 부여해주는 ID
* `[국가코드]-[조직 식별자]-[설정된 번호 조합]`중 [설정된 번호 조합]을 Case ID라고 의미한다.
* 해당 Case ID는 Case Edit의 Case ID에 표시되며 Tracker의 Case ID에 표시된다.

# Case ID 및 C1.1, C.1.8.1(자발 보고의 경우) 기본 구조 및 국가 코드 로직

* 기본 구조: `[국가코드]-[조직 식별자]-[설정된 번호 조합]`
* 국가 코드 결정 우선순위:

  + 국가코드는 케이스 정보의 국가코드를 불러와 Case ID 생성
  + 시스템은 아래 순서에 따라 접두사(Prefix) 국가 코드를 자동 결정함.

    1. Reporter's Country Code (Primary) 원보고자의 국가코드 : 보고서 내 주 기입 국가 코드
    2. Reporter's Country Code (First) 원보고자의 국가코드 : Primary가 없을 경우, 리스트의 첫 번째 국가 코드
    3. Identification of the Country Where the Reaction / Event Occurred (Primary) 발현당시 환자가 있던 국가: 위 정보가 없을 경우, 해당 필드의 Primary 국가 코드

       1. Identification of the Country Where the Reaction / Event Occurred (First) 발현당시 환자가 있던 국가: 해당 필드가 여러개 값이 있을 경우 리스트 중 첫 번째 국가 코드
    4. 기본값: 모든 조건 미충족 시 'KR' 기입
* 조직 식별자 입력:

  + 사용자가 직접 텍스트로 입력 가능

    - 최소 1글자 최대 10글자까지 설정 가능, 영어, 숫자만 입력 가능

### 번호 조합 구성 요소 설정

* Year, Type of Report, 보고 구분, 자유 텍스트, 일련번호의 조합으로 생성됨

#### Year (연도 표시 및 리셋 정책)

* 최초 발생인지일 (C.1.4)의 데이터값으로 참조
* 연도 설정은 Case ID 내에 연도 정보를 어떤 형식으로 포함할지와 번호 초기화 여부를 동시에 결정

|  |  |  |  |
| --- | --- | --- | --- |
| **구분** | **표기 방식** | **번호 초기화** | **비고** |
| 2-digit year | 최초 발생인지일 (C.1.4) 연도의 뒤 두 자리 표기 `YY` (예: 26) | 연 단위: 매년 1월 1일 00:00:00 리셋 | 연간 누적 채번 |
| 4-digit year | 최초 발생인지일 (C.1.4) 연도의 전체 네 자리 표기 `YYYY` (예: 2026) | 연 단위: 매년 1월 1일 00:00:00 리셋 | 연간 누적 채번 |
| N/A | 연도 미표기 | 누적 증가: 초기화 없이 계속 누적 | 전체 누적 채번 |

* 연도 설정란 옆에 툴팁 제공

  + "연도 선택 시 매년 초에 일련번호가 '00001'로 초기화됩니다. N/A 선택 시(Chip을 추가하지 않았을 경우) 초기화 없이 번호가 계속 누적됩니다."

#### Type of Report (보고 구분) 추가

케이스의 Type of Report 필드 데이터 값에 따라 고유한 구분자를 자동으로 추가

* 표시 문구: "체크 시 케이스의 Type of Report 값에 따라 구분자가 Case ID에 자동 추가됩니다."
* 코드 매핑 로직:

  + Spontaneous → SP 추가
  + Report from Study → RS 추가
  + Other → OT 추가
  + Not Available to Sender(Unknown) → UN 추가
* Preview에 표시: SP

#### Product Abbreviation 추가

* 표시 문구: Admin의 Product Config에 등록된 제품 약어를 참조합니다. 값이 없으면 UNK으로 표시됩니다.
* 데이터 소스: Admin 내 `Product Config`에 등록된 Product Abbreviation(제품 약어) 필드값을 참조함.
* 노출 조건: 케이스 생성 시 '제품 불러오기'를 통해 제품 정보가 케이스 내에 포함된 경우에만 해당 값이 활성화됨. 값 자체가 없을 경우 UNK으로 표시
* Suspect이자 자사 제품이 두개 이상일 경우: 제일 첫번째 순서의 자사제품 약어를 표시한다.
* Preview에 표시: PRD

#### 자유 텍스트 추가

* Free Text를 체크박스 형태로 선택 가능
* 선택 시 [Custom Text] 블록 제공.
* 작동 방식:

  + 라디오 버튼 선택하여 텍스트 블록 추가.
  + 사용자가 원하는 문자(영문, 숫자만)를 직접 입력 (예: `SELTA`, `STUDY`). 최대 6글자 입력 가능
  + 입력된 텍스트는 드래그 앤 드롭을 통해 원하는 위치에 배치 가능.

### 일련번호 자릿수 설정

* 선택값: 3, 4, 5, 6, 7, 8, 9 (최대 9자리)
* 작동: 선택한 자릿수만큼 번호 앞자리에 '0'을 채움(Zero-padding).
* 표시문구: "선택한예를 들어, 이 값을 5로 설정할 경우 숫자 앞에 0이 붙어 00001, 00002와 같은 5자리 숫자로 생성됩니다." 메시지 노출
* 중간에 케이스 자릿수 변경 시, 이후 생성되는 케이스부터 적용

### 일련번호 및 카운팅 시작점 설정

#### 시작 번호 설정

매뉴얼 관리에서 자동 채번으로 전환하거나, 내부 차수 관리가 필요한 경우 번호의 시작점을 지정합니다.

* 시작 지점 설정값은 디폴트로 1입력 (1 입력 시 1부터 시작)

  + 값이 없을 경우 강제로 1 부여: 지우고 없는 값에서 커서 아웃할 경우 강제로 1기입
* 설정 필드: Start From (시작 번호 입력란)
* 작동 로직:

  + 사용자가 숫자 N을 입력하면, 시스템은 N부터 카운팅을 시작함. (ex. 시작 번호에 25 입력 + 자릿수 5자리 설정 ➔ 첫 번째 Case ID는 00025으로 생성.)
* 적용 시점: 설정 저장 후 생성되는 첫 번째 케이스부터 즉시 적용

#### 최후 번호 검증 규칙

* Case ID 부여 시 생성 시작 번호는 해당 조합에서 이미 생성된 가장 최후의 번호보다 커야 함

  + 예: 3자리 설정 후, 최대 자리수를 003으로 설정했다 하더라도 생성된 최후 자리의 번호가 006이면 생성 시 007로 생성됨
* 검증 범위: 동일 조합 내에서 생성된 이력에 한함.

#### 안내 문구 표시

* 표시 문구: 입력한 시작번호부터 보고자 관리번호가 생성됩니다. 예를 들어, 시작 번호를 '25'로 입력하고 자릿수를 '5자리'로 설정하면, 첫 번째 케이스 ID는 '00025'와 같은 형식으로 생성됩니다. 또한, 케이스 생성 시 기존 생성된 번호 이후의 숫자만 생성됩니다.

#### 시작 번호 설정 제한 정책

* 최대 번호 설정 제한: 일련번호 고달을 방지하기 위한 하드웨어적 임계치 설정

  + 제약: 시작 번호는 [최대 생성 가능 숫자 - 400]을 초과하여 설정할 수 없음

### 포맷 설정 및 인터랙션 (Drag & Drop)

사용자가 번호 구성 요소의 순서를 직관적으로 지정하는 영역

* 표시 문구: 칩을 드래그하여 순서를 변경하면 상단 Preview에 실시간 반영됩니다.
* 노출 조건: Year를 연도(2/4-digit)로 선택하거나, Type of Report를 체크할 경우에만 해당 설정 영역이 활성화됨.
* 컴포넌트: Chip 형태의 블록(YY/YYYY, ######, TR)으로 구성됨.
* 작동:

  + 해당 영역에서 Chip을 추가하면 해당 칩 반영
  + Chip이 추가되면 하단의 해당 영역이 활성화됨

    - Chip이름: 연/월=Year, ####=일련번호, TR=보고 구분, PRD=제품 약어, TXT=자유 텍스트
    - 칩 선택하면 순서 변경 가능한 영역에 칩 반영
    - x 누르면 chip 삭제 가능
  + 블록을 드래그하여 좌우 순서를 변경하면 상단의 Case ID Preview에 실시간 반영됨.
* 예: YYYY - ##### - TR 순서 배치 → KR-SELTA-202500001TR

UX/UI 요구사항

* 포맷 설정 인터렉션은 프리뷰 하단, 설정영역의 상단에 부탁드립니다.

## Case ID 만료 경고 및 자동 자릿수 확장

### ID 팝업 표시 시점

* Case가 생성되어 Case ID가 새로 부여되는 시점에서 해당 팝업 표시 (Case ID configuration 화면에서 뜨는 화면이 아님)

  + 표시되는 시점: Manual Intake와 Import에서 케이스 생성 완료 시 표시 - 케이스 생성 완료되어 Case ID가 부여된 상황이면 어떤 화면에 있어도 해당 팝업이 표시되어야 함

### Case ID 만료 임박 알림

일련번호가 설정된 자릿수의 최대치에 도달하기 전, 사용자에게 미리 알림을 제공하여 관리적 대비할 수 있도록 안내

#### 팝업 노출 조건

* 임계치: 남은 번호가 100개 이하일 때 팝업을 노출
* 노출 시점: 케이스에 Case ID가 최종적으로 부여된 순간, 남은 번호가 100개 이하가 되면 즉시 팝업으로 안내
* 기준 설정 사유: 시스템이 한 번에 최대 30개까지 병렬 처리가 가능하므로, 동시 사용자가 대량의 작업을 수행할 때 안내를 놓치지 않도록 충분한 버퍼인 100개를 기준으로 지정
* 팝업 내용: "현재 마지막으로 부여된 케이스 ID는 [Case ID]입니다.최대 자릿수[Case ID 최대값]에 도달하면 번호 체계가 자동으로 확장되어 [New Case ID\_1],[[New Case ID\_2] 순으로 계속해서 부여됩니다.번호 중복이나 작업 중단 없이 정상적으로 처리되니 안심하고 사용하시기 바랍니다."

#### 사용자 인터랙션 및 제어

* 대상: 해당 시스템을 사용하는 모든 사용자에게 노출
* 메시지 내용: 현재 마지막으로 부여된 최신 Case ID를 함께 노출하여 상태를 인지시킴
* 닫기 로직:

  + 단순 [Confirm] 클릭: 팝업이 닫히지만, 해당 사용자가 다음번 로그인 시 다시 노출
  + [오늘 하루 보지 않기] 체크 후 [Confirm] 클릭: 해당 팝업을 하루동안 노출하지 않음
  + 자동 미노출: 일련번호가 완전히 초과되어 새로운 자릿수 체계(1XXXXX...)로 넘어간 경우에는 기존 팝업 노출을 중단

### Case ID 자릿수 자동 확장

설정된 자릿수를 모두 소진하더라도 서비스 중단 없이 시스템이 자동으로 자릿수를 확장하여 번호를 부여

#### 자릿수 확장 동작

* 동작: 설정된 최대 자릿수(예: 5자리)를 초과하는 번호가 필요한 경우, 자릿수를 한 자리 추가하여 번호를 생성
* 번호 체계: 확장된 자릿수에서 '1XXXXX' 형식으로 번호를 시작

#### 자릿수 확장 예시

* 기본 설정: 시리얼 넘버 길이를 5로 설정한 경우

  + 최종 번호: `99999`
  + 초과 시 첫 번호: `100000` (6자리로 자동 확장)
  + 이후 진행: `100001`, `100002` 순으로 `999999`까지 정상적으로 부여

* 적용 범위: 해당 설정은 오직 'Import' 기능을 통해 케이스가 생성될 때만 적용되는 로직 (추후 Convert에도 적용 예정)

## C.1.1, C.1.8.1 동일 입력 설정

### C.1.8.1이 없을 때 C.1.1 값을 그대로 사용

* 동작: C.1.8.1 값이 비어 있는 경우(Null)에 한해, C.1.1의 번호를 그대로 C.1.8.1에 복사하여 기입
* 목적: 기존에 부여된 고유 식별자가 있다면 보호하고, 없는 경우에만 자동 생성하여 편의성을 높임.

# ICSR 식별자 관리 사양서

## Case vs Report Identifier 노출 사양

사용자가 보는 화면의 성격에 따라 노출되는 형식을 다르게 적용

|  |  |  |  |
| --- | --- | --- | --- |
| **구분** | **노출 위치** | **표시 형식** | **비고** |
| Case Level | Tracker List, Case의 Case ID | 일련번호 | 예: `202604ST001`, `202604ST00002` |
| Report Level | ICSR 보고서 (C.1.1, C.1.8.1) | 국가-조직-일련번호 | 예: `KR-SELTA-202604ST001` |

## 생성 유형별 식별자 부여 로직

케이스가 시스템에 인입되는 방식에 따라 C.1.8.1의 생성 규칙이 결정

### 매뉴얼 케이스 생성 (Initial)

* Case ID: Configuration 설정에 따른 순차적 일련번호 부여.

  + **해당 Case ID가 식별자로 사용.** 중복 체크 및 후속 보고(Follow-up) 연결의 기준으로 사용함.
* Report ID (C.1.1 / C.1.8.1):

  + C.1.1과 C.1.8.1이 동일
  + 보고서의 Destination(보고처)이 확정되는 시점에 국가 코드(예: KR)가 결정됨.
  + [국가]-[회사]-[일련번호] 조합으로 생성되어 보고서에 기입됨.

### 데이터 인입 (Import / AI Convert(추후))

* Case ID: Configuration 설정에 따른 순차적 일련번호 부여.

  + **식별자: 인입된 파일 내의 C.1.8.1 값을 기반으로 생성됨. 중복 체크 및 후속 보고(Follow-up) 연결의 기준으로 사용함.**
* Report ID (C.1.1 / C.1.8.1):

  + C.1.1

    - 보고서의 Destination(보고처)이 확정되는 시점에 국가 코드(예: KR)가 결정됨.
    - [국가]-[회사]-[일련번호] 조합으로 생성되어 보고서에 기입됨.
  + C.1.8.1

    - 파일 내 기재된 기존 C.1.8.1 값을 그대로 유지
    - 이 경우 케이스의 식별값과 보고서의 식별값은 동일한 `C.1.8.1`을 공유함

## 식별자 수정 및 변경 정책

데이터 무결성 확보를 위해 "한 번 부여된 식별자는 수정할 수 없다"는 원칙을 적용

* 수정 불가: 시스템에 의해 부여되는 즉시 수동 수정이 절대 불가함. (사용자 편집 권한 없음)

C.1.1 변경 시점:

* C.1.1은 Initial 사례에만 부여.
* C.1.1은 개별 수정

  + Approval 상태 이후 번호 더이상 변경 불가

    - UI: 수정 버튼 비표시.
    - 문구: Approval 상태 이후로는 보고자 관리번호(C.1.1)과 고유식별 보고자 관리번호(C.1.8.1) 수정이 불가합니다.
