# 1. 알림 정책

본 섹션은 플랫폼의 인앱 알림 및 이메일 알림 동작 방식을 정의한다.

## 1.1 알림 유형

알림은 일반 알림과 진행 상태 알림 두 가지로 분류된다. 시스템이 정상적으로 동작하고 있다는 점을 사용자에게 명확하게 인지시키기 위한 목적으로 두 유형을 구분한다.

|  |  |  |
| --- | --- | --- |
| 유형 | 설명 | 발신 기준 |
| 일반 알림 | 완료된 작업 결과, 설정 변경, 오류 발생 등 결과 중심 메시지 전달. 단일 이벤트의 결과를 나타내며 트리거 이벤트는 사용자 또는 시스템 자동 처리에 의해 발생 | 완료 / 오류·실패 |
| 진행 상태 알림 | 변환, 보고와 같은 진행 중인 작업의 상태를 실시간으로 전달하여 시스템 활동의 가시성을 확보 | 진행 중 상태 변화 |

## 1.2 알림 채널

|  |  |  |
| --- | --- | --- |
| 채널 | 기본 여부 | 적용 범위 |
| 인앱 (In-App) | 기본 | 모든 알림 종류. 앱 접속 중 실시간 수신 |

## 1.3 알림 범위 (역할별)

알림 수신 범위는 역할 및 접근 권한에 따라 결정된다.

|  |  |
| --- | --- |
| 역할 | 알림 범위 |
| Master Admin | 전체 Org 범위 시스템 알림 + 본인 관련 워크플로우 알림 |
| Org Admin | 해당 Org 범위 시스템 알림 + 본인 관련 워크플로우 알림 |
| App Admin | 접근 가능한 앱 범위 알림 + 본인 관련 워크플로우 알림 |
| App User | 접근 가능한 앱 범위 알림 + 본인 할당 워크플로우 알림 |
| External User | Safety DB 앱 범위 알림 + 본인 할당 워크플로우 알림 |
| Viewer | 해당 Org 범위 시스템 알림 + 본인 관련 워크플로우 알림 |

참고 App Admin / App User는 Org 레벨에서 Org User에 해당

## 1.4 인앱 알림 UI 구조

사이드바 알림 아이콘을 통해 접근하는 드롭다운 패널로 구성된다.

|  |  |
| --- | --- |
| 요소 | 설명 |
| 알림 아이콘 | 사이드바에 알림 아이콘으로 표시. 새로운 알림이 있을 경우 빨간 배지로 표시. 해당 알림 클릭 시 읽음 처리 |
| 패널 헤더 | "알림" 타이틀 + [Mark all as read] 버튼 |
| 알림 항목 | 날짜 / 알림 카테고리 / 알림 메시지. 클릭 시 해당 화면 또는 상세 페이지로 이동 |
| 표시 건수 | 최대 20건. 21건부터는 표시하지 않음, 알림 0건 시 패널 내 "새로운 알림이 없습니다." 표시. 배지 숨김. |
| 정렬 | 최신 순(내림차순) 고정 |

*[Mark all as read] 클릭 시 "모든 알림을 읽음으로 처리합니다. 계속 하시겠습니까?" 확인 팝업 표시. [Confirm] 클릭 시 일괄 읽음 처리, [Cancel] 클릭 시 팝업 닫힘.*

### 알림의 실시간 갱신

\*알림 패널이 열려있는 상태에서 새 알림 도착 시 패널에 즉시 추가 표시.

\*배지 카운트는 새 알림 도착 시 즉시 갱신.

-> 해당 기술에 대해서는 작성은 하되, 현재 개발팀의 설계에 따른다.

### 📋 알림 패널 열기 흐름

|  |  |
| --- | --- |
| 단계 | 내용 |
| ① | 사이드바 알림 아이콘 클릭 → 알림 드롭다운 패널 열림\* |
| ② | 최근 알림 최대 20건 표시. 미읽음 항목은 강조 표시 |
| ③ | 패널 외부 클릭 → 패널 닫힘 |

\*참고: 알림 클릭 시 해당 이벤트 화면으로 이동한다. 기존 탭 재사용 여부 및 자동 포커스 이동은 보장하지 않고, 세부 탭 처리는 개발 설계를 따른다.

### 📋 알림 읽음 처리 흐름

|  |  |
| --- | --- |
| 단계 | 내용 |
| ① | 알림 항목 클릭 → 해당 이벤트 화면으로 이동 + 개별 읽음 처리 |
| ② | 이동 후 배지 카운트 감소. 0건이 되면 배지 숨김 |
| ③ | [Mark all as read] 클릭 → 확인 팝업 표시 ("모든 알림을 읽음으로 처리합니다. 계속 하시겠습니까?") |
| ④ | [Confirm] 클릭 → 전체 알림 일괄 읽음 처리. 배지 즉시 초기화. [Cancel] 클릭 → 팝업 닫힘. 읽음 처리 없음 |

## 1.5 알림 상태 관리

|  |  |
| --- | --- |
| 항목 | 규칙 |
| 미읽음 배지 | 미읽음 알림 수 표시. 0건 시 배지 숨김 |
| 읽음 처리 | 알림 항목 클릭 시 개별 읽음 처리. [Mark all as read] → 확인 팝업 후 일괄 처리 |
| 인앱 보존 기간 | 수신 후 2주. 2주 경과 시 알림창에서 자동 삭제, 최대 100개 보관 |
| 표시 건수 제한 | 최대 20건 표시. 21건부터는 표시하지 않음 |
| 알림 정렬 | 최신 순(내림차순) 고정 |
| 중복 알림 | 동일 이벤트에 대해 중복 발송하지 않는다 |
| 탭 간 읽음 동기화 | 한 탭에서 읽음 처리 시 동일 브라우저 내 모든 탭에 즉시 반영. 배지 카운트 실시간 갱신. |

## 1.6 알림 종류 정의

알림은 카테고리별로 분류된다. 아래 표는 현재까지 정의된 일반 알림 항목이다. 각 기능 기획이 추가될 때 해당 기능의 알림 항목을 이 표에 추가하는 방식으로 관리한다.

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 카테고리 | 수신 대상 | 트리거 | 알림 메시지 (예시) | 클릭 이동 |
| Account | 해당 계정 | 권한 변경 안내 | 계정 권한이 [새 권한명](으)로 변경되었습니다. | 이동 없음 |
| Connected Service | Master Admin/Org Admin | 연결 서비스 라이선스 기간 만료 시 | [WHODrug license no/MedDRA license no]가 만료되었습니다. | 해당 태넌트의 어드민 > 라이센스 관리 페이지로 이동    단, 이동은 알림을 Org 홈에서 확인한 경우에만 이동 가능하고, 워크스페이스 홈/앱에서는 알림은 확인 가능하나 이동은 불가 |
| Compliance | 해당 계정 | 할당된 케이스의 due date가 3일 남았을 시 (3일 전부터 매일 00:00~01:00 사이 발송) | [Case ID]의 보고 기한이 3일 남았습니다. | 해당 워크스페이스의 트래커로 이동 |
|  | 해당 계정 | 할당된 케이스의 due date가 당일일 경우 | [Case ID]의 보고 기한이 오늘까지입니다. | 해당 워크스페이스의 트래커로 이동 |
|  | 해당 계정 | 보고 시작 | [Case ID]의 보고가 시작되었습니다. | 해당 워크스페이스의 보고 페이지로 이동 |
|  | 해당 계정 | 보고 완료 | [Case ID]의 보고가 성공적으로 완료 되었습니다. | 해당 워크스페이스의 보고 페이지로 이동 |
|  | 해당 계정 | 보고 실패 | [Case ID]의 보고에 실패했습니다. | 해당 워크스페이스의 보고 페이지로 이동 |
| Workflow | 해당 계정 | assignee로 할당되었을 시 | [Case ID]가 할당되었습니다. | 해당 워크스페이스의 트래커로 이동 |
|  | 해당 계정 | 내가 오너인 케이스의 상태가 변경되었을때 | [Case ID] 상태가 변경되었습니다. [변경 전 상태] -> [변경 후 상태] | 해당 워크스페이스의 케이스 상세 페이지로 이동 |
