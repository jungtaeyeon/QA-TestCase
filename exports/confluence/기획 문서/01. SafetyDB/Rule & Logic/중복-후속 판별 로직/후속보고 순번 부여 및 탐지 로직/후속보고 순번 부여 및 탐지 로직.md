# 후속보고 순번(SEQ) 부여 로직 설계서

## 설계 개요

* **목적**: 데이터 소스별 특성에 맞춰 후속보고의 순번(SEQ)을 자동 부여함으로써 타임라인 기반의 사례 이력 관리 및 버전 비교의 기준값을 확보함.
* **핵심 원칙**: 모든 후속보고 데이터는 '최근 인지일(C.1.5)' 정보를 반드시 포함하여 정렬의 기준점으로 삼음.

## 순번 부여 프로세싱 로직

시스템은 데이터 소스의 종류에 따라 아래 세 가지 분기 로직을 적용하여 순번을 확정

### 자사 수집 사례

보고 무효화/수정, Report Nullification/Amendment(C.1.11.1) 필드의 값에 따라 아래 로직을 따른다.

* 최초(Initial): 동일한 식별자가 없는 경우 ➔ '0' 부여.
* 후속(Follow-up):

  + 수정/무효화(Amend/Null)이 아닐 경우:

    - 동일 식별자 사례에 대해 아래 우선순위로 정렬 후 1부터 순차 부여.

      * 인지일 (C.1.5) : 최우선
      * 시스템 생성 일시 (C.1.2)
  + 수정/무효화(Amend/Null)일 경우:

    - 이전 보고와 동일한 번호를 유지하여 부여함. (ex. Fu No1이후에 amend일 경우 1amend로 표시)

### 재전송 사례 (Resubmitted Case)

* 최초(Initial): 동일한 C.1.8.1이 존재하지 않는 경우 ➔ '0' 부여.
* 후속(Follow-up):

  + 수정/무효화(Amend/Null)이 아닐 경우:

    - 동일 C.1.8.1 사례에 대해 아래 5단계 우선순위로 정렬 후 1부터 순차 부여.

      * 인지일 (C.1.5): 최우선
      * 시스템 생성 일시 (C.1.2)
      * 이전 전송자 인지일 (Original C.1.5)
      * 원본 보고서 생성일시 (Original C.1.2)
      * 파일 내 나열 순서 (In-File Order)
  + 수정/무효화(Amend/Null)일 경우:

    - 이전 보고와 동일한 번호를 유지하여 부여함

## 최종 출력 및 활용

* 최종 부여 순번 (Final SEQ Number): 확정된 SEQ 값은 각 케이스의 메타데이터로 영구 저장됨.
* 활용 지표: 사례별 이력 관리의 타임라인을 구성하고, 서로 다른 버전 간의 데이터를 비교하는 핵심 기준값으로 활용함.
