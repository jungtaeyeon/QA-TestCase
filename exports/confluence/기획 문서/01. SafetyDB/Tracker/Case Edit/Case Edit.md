# 개요

* 목적: SafetyDB(iVigilance Square)의 핵심 데이터 입력 화면인 Case Edit 페이지의 공통 구성 요소, 화면 배치 및 시스템 제어 로직을 정의. 사례의 고유 정보를 상시 노출하고, 섹션 기반의 데이터 입력과 업무 종결/삭제 액션을 효율적으로 수행할 수 있도록 설계.
* 페이지 위치: Safety > Tracker > Case Edit
* Case Edit 내 MFDS 모든 필드의 Validation은 [MFDS 약물이상반응 및 이상사례 개별 항목 검증 룰(링크)](https://seltaglobal.sharepoint.com/%3Ax%3A/r/sites/RDDivision/_layouts/15/Doc.aspx?sourcedoc=%7BE96CEE4B-659D-46AE-92BF-B1DBDDC6CBD6%7D&file=%EC%95%BD%EB%AC%BC%EC%9D%B4%EC%83%81%EB%B0%98%EC%9D%91%20%EB%B0%8F%20%EC%9D%B4%EC%83%81%EC%82%AC%EB%A1%80%20%EA%B0%9C%EB%B3%84%20%ED%95%AD%EB%AA%A9%20%EA%B2%80%EC%A6%9D%20%EB%A3%B0(20240124).xlsx&action=default&mobileredirect=true&DefaultItemOpen=1%3Fweb%3D1) 을 따른다. 추가로 로직 및 추가 검증룰이 필요한 경우는 기재된 상세 사항을 따른다.

# 4. 정책

---

## 수정 잠금 정책 (Lock)

업무 단계가 Approved 및 이후의 단계인 경우 모든 필드는 Read-only 처리. 수정을 위해서는 라우팅을 통해 단계를 이전(Entry, QC 등)으로 돌려야 함.

|  |  |
| --- | --- |
| 조건 | 동작 |
| 업무 단계가 Approved 이후 | 모든 필드 Read-only |
| 수정이 필요한 경우 | Unlock 버튼을 통해 전단계로 Return하고 수정 가능 |
| 타 사용자가 이미 편집중일 경우 | ‘[User Name]이 현재 편집 중입니다. 읽기 모드로 진입하시겠습니까?’ 표시. Confirm/Cancel 버튼. Confirm시 읽기 모드로 진입. Cancel시 Tracker 페이지에 머뭄 |

## 수동 수정 불가 필드

Case ID, FU 정보, C.1.1, C.1.8.1 필드는 사용자가 직접 수정할 수 없으며 시스템이 자동 관리.

## 그룹 동기화

묶여 있는 모든 FU 사례는 상단 헤더의 식별자 정보를 공유하며, 한 사례의 식별자 정보가 변경될 경우 그룹 전체가 동기화.

## FU 자동 생성 로직

* 데이터 계승: 기존 사례의 모든 정보가 동일하게 기재된 상태(복사)로 생성
* 식별자 상속: 동일한 Case ID 및 식별자(Identifier) 체계를 공유
* FU 번호: 시스템 로직에 따라 업데이트 (▶ 후속보고 순번 부여 및 탐지 로직 참조)
* 결과: 신규 FU 사례 생성 완료 시 해당 사례의 Case Edit 화면으로 즉시 전환

## 케이스 삭제 정책

|  |  |
| --- | --- |
| 조건 | 삭제 가능 여부 |
| 보고 이력이 없는 초기 오류 건 | 가능 |
| 보고 이력 존재 | 불가 |
| 타 사용자 편집 중 | 불가 |

### 삭제 시 삭제 사유 기제 모달 표시

* ‘케이스를 정말 삭제하시겠습니까? 삭제할 경우, 케이스를 삭제하려는 사유를 입력해주세요.’

  + 버튼: Cancel/Delete. Delete 클릭 시 삭제

## 케이스 완료 정책

**케이스 완료 (Case Complete)**

* 작동: 버튼 클릭 시 "케이스를 완료하시겠습니까?"라는 확인 모달을 노출
* 결과: Confirm 클릭 시 해당 사례의 ICSR 단계를 즉시 Completion으로 변경하며, Tracker의 단계 컬럼에도 즉시 반영

## 미저장 변경사항 경고

* 화면설계도(p56)에서는 Processing 중 Save하지 않고 페이지를 벗어나는 모든 행위(타 메뉴 클릭, 웹 페이지 닫기 등) 시

  + "저장하지 않고 페이지를 벗어날 경우, 변경한 내용을 잃을 수 있습니다. 이동하시겠습니까?" 확인 팝업을 표시. Cancel/Confirm 버튼 표시

## Validation 맞지 않을 경우 문구 표시

* 검증 룰과 맞지 않을 경우 validation 문구 표시

  + ‘유효하지 않은 값입니다. 다시 입력해주세요’

# 5. 요구사항

---

|  |  |  |  |
| --- | --- | --- | --- |
| 요구사항 | 기획 방향 | 의도 | 분류 |
| 사례 식별 정보 상시 노출 | 상단 고정 헤더에 Case ID, FU, C.1.1, C.1.8.1 상시 표시 | 스크롤 시에도 주요 정보와 액션 버튼을 놓치지 않도록 함 | Requirement |
| FU 생성 | [+] 버튼으로 기존 사례 복사 기반 FU 생성 | 후속보고 생성 시 반복 입력을 제거하고 데이터 일관성 확보 | Requirement |
| 통합 저장 | [Save] 버튼으로 모든 섹션 데이터 일괄 저장 | 섹션별 개별 저장이 아닌 통합 저장으로 데이터 정합성 보장 | Requirement |
| 케이스 잠금 해제 | [Unlock] 버튼으로 단계를 재설정하고 케이스 잠금 해제 | 케이스가 수정 필요한 상황에서 Lock을 해제하고 수정하기 위해. 사유가 필요하므로 사유 기재 필요 | Requirement |
| 케이스 완료 처리 | [Case Complete]로 Completion 단계 전환 | 업무 완료 시점 명확화 및 Tracker 즉시 반영 | Requirement |
| 케이스 삭제 | [Delete]로 확인 후 삭제 및 Tracker 리다이렉트 | 오류 건 정리 및 데이터 정합성 확보 | Requirement |
