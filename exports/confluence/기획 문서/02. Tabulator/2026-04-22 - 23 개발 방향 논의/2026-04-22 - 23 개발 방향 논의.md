## 1. 솔루션의 목적과 정체성

### 1.1 한 줄 정의

> **Tabulator는 안전성 Data-Set에서 PV Aggregate Report 작성을 위한, 데이터 가공·분석 도구이다.**

### 1.2 정체성의 명확화

기존 Tabulator는 데이터 조회(Tracker), 실시간 대시보드, ICSR 상세 조회 등 다양한 기능이 혼재되어 있어 **Safety 앱과의 역할 구분이 모호**한 상태임. 이번 리뉴얼에서는 다음 원칙에 따라 정체성을 좁혀서 정의.

* **Tabulator는 DB가 아니다.** 케이스 영속 보관 책임은 Safety가, 참조용 raw data 보관 책임은 (도입 시) Intake Hub가 가진다.
* **Tabulator는 일회성 분석 엔진이다.** Task 단위로 데이터를 입력받아 → 가공·검증 → 산출물을 생성하고, raw data는 Task lifecycle 내에서만 유지한다.
* **Tabulator는 표 생성의 단일 책임자이다.** Safety 안에 표 생성 기능을 두지 않고, 모든 Aggregate Report용 line-listing과 표 생성은 Tabulator에서 처리한다 (대표님 결정 사항).

**cf. 원 프로젝트 의도와의 충돌** 🔵

* **원 프로젝트 의도**: Tabulator를 Safety/IB 쪽에 더 가깝게 통합하여 데이터 관리 도구로 확장
* **검토 후 제안**: 정체성을 좁혀 Task 단위 일회성 가공 도구로 명확화

| **결정 상태** | 🟢 QC 강화 방향 합의 / 🟡 구체적 검증 항목 범위는 제안 단계 |

### 2.4 데이터 관리 방식

| 구분 | 내용 |
| --- | --- |
| **요구사항** | KAERS data를 자사 DB에 영속 저장하길 원치 않는 고객사 다수. 별도 데이터 허브에 대한 니즈. 클라우드 비용과 고객사 원시자료 분리 필요. |
| **분석** | 외부 출처 데이터(KAERS, JnJ 등)는 자사 인지 케이스가 아니므로 보고 의무가 없지만, 조직 차원의 분석에는 필요하다. 이 둘을 같은 DB에서 다루면 과금·책임 경계가 모호해진다. |
| **개선안 A** 🟡 | **Tabulator는 자체 DB를 갖지 않음.** Task 단위 작업 후 raw data는 폐기. |
| **개선안 B** 🔵 | **Intake Hub 신설** (별도 프로젝트). 모든 raw 데이터는 Hub에 보관, Safety로의 promote는 사용자 컨펌. |
| **결정 상태** | 🔵 두 개선안 모두 추가 의사결정 필요. 특히 Intake Hub는 본 리뉴얼 범위 밖에서 별도 트랙 추진 권고. |

### 해당 방식 적용 시 IVI SQUARE 플랫폼 구성 예상안

```
IB Studio (플랫폼)
├── Intake Hub (가벼운 DB) ......... [신규 도입 검토 사항]
│     └─ 모든 raw 이상사례 데이터 (line-listing 수준, 수정 불가)
│     └─ 출처: 직접 업로드 / KAERS / Literature / Partner data
│
├── Safety (무거운 DB)
│     └─ 컨펌된 ICSR 케이스 (편집·보고 가능)
│
├── Tabulator (분석 엔진, 자체 DB 없음)
│     └─ Task 단위 작업, 산출물·작업 메타데이터만 보관
│
└── Sync, Health 등 기타 유닛
```

### 2.5 과금 모델

| 구분 | 내용 |
| --- | --- |
| **요구사항** | 현재 Generate 시점 차감은 사용자 실수에 과도한 비용 부담. 같은 파일 반복 Generate 시 무한 차감 또는 무한 재생성 등 양극단 모순 발생. |
| **분석** | PV 실무에서 데이터 클렌징은 본질적으로 반복 작업(MedDRA 재코딩, listedness 재판정 등)이다. Generate 단위 과금은 이 본질과 충돌한다. Confirm = "이 산출물을 규제 보고에 사용하겠다"는 의사 확정 시점이며 비용 발생에 가장 자연스러운 시점이다. |

| **결정 상태** | 🟢 차감 시점·단위 합의 / 🔵 동일 Task 재생성, 후속 데이터 추가 작성 등 세부 정책은 마케팅과 협의 필요 |

## 3. 제안하는 Workflow 🟡 [제안]

### 3.1 Task 단위 7단계

| 단계 | 명칭 | 핵심 행위 | 산출물 |
| --- | --- | --- | --- |
| 1 | **데이터 입력** | 직접 업로드 또는 (Phase 2) Safety/Hub에서 끌어오기, MedDRA 버전 선택 | 원본 데이터셋 |
| 2 | **분석 유형 결정** | Renewal / PBRER / (향후) DSUR / Custom 중 선택, 보고서 요건 입력(DLP, 기간 등) | 분석 설정값 |
| 3 | **데이터 범위 확정** | 분석 요건에 따라 필터링, 포함/제외 케이스 확정 | 분석 대상 데이터셋 |
| 4 | **데이터 검증·수정** | MedDRA recoding(우선), 필수 필드 검증, 향후 listedness 등 확장 | 검증 완료 데이터셋 |
| 5 | **미리보기 (Line-listing + 표)** | 화면에서 QC, 셀 클릭 시 원본 case로 drill-down | QC 완료 draft |
| 6 | **Confirm** | 사용자 최종 확정 (이 시점에 품목 수 기준 과금 발생) | 확정된 작업 |
| 7 | **결과물 생성·다운로드** | 정식 산출물 생성 (Excel/PDF 등), Library에 저장 | 최종 산출물 |

### 3.2와 3 단계 순서 결정 필요 🟡

분석 유형(2단계)과 데이터 범위 확정(3단계)의 순서 결정 필요: 아래 분석 유형, 단, 현재는 Renewal Report만 해당하므로, 추후 고도화

* **Renewal Report**: 국내 시판 후 6개월/1년 단위 수집 데이터, 한국 KAERS 중심
* **PBRER**: International Birth Date(IBD) 기준 글로벌 데이터, DLP 기준 cut-off
* **DSUR**: Development International Birth Date(DIBD) 기준, 임상 데이터 중심

보고서 종류에 따라 필요 데이터가 다르므로, 어떤 보고서를 쓸지가 먼저 정해져야 어떤 데이터를 가져올지가 결정되는지.

### 3.3 Workflow 운영 정책 🟡 [제안]

* **되돌아가기**: Confirm 전(1~5단계)까지는 모든 단계로 자유 이동 가능. Confirm 후에는 새 Task로 재시작.
* **임시 저장 단위**: Task 자체가 단위. 작업 중인 Task는 "Draft" 상태.
* **Drill-down**: 5단계 미리보기에서 표의 특정 셀 클릭 시 원본 case로 즉시 이동.
