#1C.1.1 수정 가능 여부 — 두 섹션이 정반대 정의

모순
섹션 A — "Case Edit에서 동작 > C.1.1 동작 설정":
"기본으로 수정 불가. 만약 수동으로 수정하고 싶다면, 1개의 단계를 거쳐 해지 가능 → 수정 버튼 클릭 → Confirm → 수동 수정으로 변경"
섹션 B — "식별자 수정 및 변경 정책":
"수정 불가: 시스템에 의해 부여되는 즉시 수동 수정이 절대 불가함. (사용자 편집 권한 없음)"

섹션 A에서는 Confirm 후 수동 수정이 가능하다고 하고, 섹션 B에서는 "절대 불가"라고 정의합니다. 개발자 입장에서 어느 정책을 따라야 할지 판단 불가합니다.

📌 출처: "Case Edit에서 동작" 섹션 vs "식별자 수정 및 변경 정책" 섹션
C.1.1은 "조건부 수정 가능" 또는 "절대 불가" 중 하나로 통일 필요. 식별자 수정 정책 섹션에 "C.1.1은 Case Edit 화면에서 수동 전환 가능 (단, Approval 이후 불가)" 등으로 예외 조건을 명시하거나, Case Edit 섹션의 수정 기능을 삭제해야 합니다.

#2일련번호 자릿수 선택 범위 — 본문 vs 와이어프레임 불일치

모순
기획서 본문 (일련번호 자릿수 설정):
"선택값: 3, 4, 5, 6, 7, 8, 9 (최대 9자리)"
와이어프레임 #5 (액션 열):
"드롭다운 선택: 5, 6, 7, 8, 9 중 하나를 선택함."

본문에는 3, 4자리가 포함되지만, 와이어프레임에서는 5자리부터 시작합니다. 어느 쪽이 정확한 스펙인지 개발자가 판단할 수 없습니다.

📌 출처: "일련번호 자릿수 설정" 본문 vs 와이어프레임 표 #5
최소 3자리(999건)인지 최소 5자리(99,999건)인지 확정하고, 본문과 와이어프레임 양쪽을 동일하게 수정해야 합니다.

#3Year 데이터 소스 — "최초 발생인지일" vs "케이스 인입 연도"

모순
Year 설정 섹션 상단:
"최초 발생인지일 (C.1.4)의 연월을 데이터값으로 참조"
Year 설정 표 내부:
모든 구분(2-digit, 4-digit, YYMM, YYYYMM)에서 "케이스 인입 연도"로 표현

"최초 발생인지일(C.1.4)"과 "케이스 인입 연도"는 전혀 다른 시점입니다. 예: 2025년에 발생한 이상반응이 2026년에 보고되면, C.1.4 기준으로는 2025, 인입 기준으로는 2026이 됩니다. 연도 리셋 및 일련번호 초기화 로직에 직접적 영향을 미치는 핵심 차이입니다.

📌 출처: "Year (연도 표시 및 리셋 정책)" 섹션
참조 시점을 "C.1.4(최초 발생인지일)" 또는 "케이스 Creation 시점" 중 하나로 확정하고, 표의 표현도 통일해야 합니다. PV 실무에서는 보고일(접수일) 기준이 일반적이므로 확인이 필요합니다.

#4Import 설정 적용 범위 — Convert 포함 여부 모순

모순
Import 설정 섹션:
"해당 설정은 오직 'Import' 기능을 통해 케이스가 생성될 때만 적용되는 로직 (추후 Convert에도 적용 예정)"
값 유지 선택 및 동기화 매트릭스(동 섹션 하위):
표에 Convert 동작이 이미 상세 정의되어 있음 (Import/Convert 모두 각 케이스별 C.1.1, C.1.8.1 동작 명시)

"추후 적용 예정"이라고 하면서 바로 아래 표에서 Convert 동작을 이미 정의하고 있어, 현재 구현에 포함해야 하는지 불명확합니다.

📌 출처: "Import 설정" 섹션 본문 vs 동기화 매트릭스 표
v1.0 범위에서 Convert가 포함인지 아닌지 명확히 기재하고, 미포함이면 매트릭스 표의 Convert 행에 "(v1.1 예정)" 등을 표시해야 합니다.
🟠 내부 불일치 (Major) — 같은 항목의 값·표현이 다름

#5번호 조합 구성 요소 목록 — 동일 항목 중복 기재

불일치
"번호 조합 구성 요소 설정" 섹션:
"Year, Type of Report, 보고 구분, 자유 텍스트, 일련번호의 조합으로 생성됨"

"Type of Report"와 "보고 구분"은 동일 항목입니다(하위 섹션 제목도 "Type of Report (보고 구분) 추가"). 구성 요소가 5종인지 6종인지 혼동됩니다. 또한 Product Abbreviation이 목록에서 빠져 있으나, 하위 섹션에는 별도로 정의되어 있습니다.

📌 출처: "번호 조합 구성 요소 설정" 섹션 첫 줄
"Year, Type of Report(보고 구분), Product Abbreviation(제품 약어), Custom Text(자유 텍스트), Serial Number(일련번호)의 조합"으로 정리 필요.

#6Case ID Preview 구분자('-') 적용 범위 모호

불일치
와이어프레임 #1:
"[국가]-[조직]-[숫자] 사이에 하이픈 표시"
포맷 설정 본문 예시:
"YYYY - ##### - TR 순서 배치 → KR-SELTA-202500001SP"

와이어프레임에서는 [국가]-[조직]-[숫자] 사이에만 하이픈이라 하면서, 포맷 설정에서는 "YYYY - ##### - TR" (각 구성 요소 사이에 '-')로 표현합니다. 그런데 실제 예시 결과인 KR-SELTA-202500001SP에서는 연도, 일련번호, TR 사이에 하이픈이 없습니다.

즉 구성요소 내부(YYYY/일련번호/TR/PRD/TXT)는 연결 사용인지, 각각 '-'로 분리하는지 불명확합니다.

📌 출처: 와이어프레임 #1 vs 포맷 설정 본문 예시
예시를 각 케이스별로 구체적으로 재작성. 예: 국가-조직-[연도][일련번호][TR][PRD][TXT] (내부 연결) 또는 국가-조직-연도-일련번호-TR-PRD-TXT (모두 분리) 중 확정 필요.

#7Case/Report 식별자 예시의 "ST" 코드 — 미정의

불일치
"Case vs Report 식별자 노출 사양" 표:
Case Level 예: 202604ST001
Report Level 예: KR-SELTA-202604ST001

Type of Report 코드 매핑은 SP, RS, OT, UN으로 정의되어 있는데, 예시에는 "ST"라는 미정의 코드가 사용됩니다. 임의 예시인지, 별도 코드인지 불분명합니다.

📌 출처: "Case vs Report 식별자 노출 사양" 표
예시를 정의된 코드(SP, RS, OT, UN)로 교체하거나, "ST"가 별도 의미가 있다면 매핑 로직에 추가 필요.
🟡 모호 / 불명확 (Minor) — 해석에 따라 구현이 달라질 수 있는 정의

#8자유 텍스트 — "체크박스" vs "라디오 버튼" 혼용

모호
"Free Text를 체크박스 형태로 선택 가능 … 라디오 버튼 선택하여 텍스트 블록 추가."

자유 텍스트 활성화 방식이 체크박스인지 라디오 버튼인지 혼동됩니다. 실제 와이어프레임/Figma는 칩(Chip) 추가 방식으로 구현되어 있어, "체크박스 형태"와 "라디오 버튼"이라는 표현 모두 현재 디자인과도 불일치합니다.

📌 출처: "자유 텍스트 추가" 섹션
실제 UI 방식(칩 추가)에 맞춰 "Segment Builder에서 'Custom Text' 칩을 추가하면 활성화됨"으로 정리 권장.

#9포맷 설정 노출 조건 — 전체 칩 vs Year/TR만 언급

모호
"노출 조건: Year를 연도(2/4-digit)로 선택하거나, Type of Report를 체크할 경우에만 해당 설정 영역이 활성화됨."

Product Abbreviation과 Custom Text도 칩 추가 시 설정 영역이 활성화되어야 하는데, 본문에서는 Year와 TR만 언급합니다. 와이어프레임 #3에서는 "칩을 추가하면 칩에 해당하는 하단의 설정 영역이 활성화됨"이라 모든 칩에 해당하는 것으로 읽히므로, 본문의 표현이 실제 의도를 축소하고 있습니다.

📌 출처: "포맷 설정 및 인터랙션" 섹션 > 노출 조건
"각 구성 요소 칩(Year, TR, PRD, TXT)을 추가하면 해당 칩에 대응하는 하단 설정 영역이 활성화됨"으로 수정 권장.

#10시작 번호 최대값 / 유효성 검증 규칙 미정의

모호
"시작 지점 설정값은 디폴트로 1입력 (1 입력 시 1부터 시작)"

시작 번호의 최대값에 대한 정의가 없습니다. 예: 자릿수 5자리일 때 시작 번호로 99999를 입력하면 첫 케이스에서 바로 만료 경고가 발생합니다. 또한 음수, 0, 소수점, 자릿수 초과값 입력 시의 유효성 검증 규칙도 명시되지 않았습니다.

📌 출처: "시작 번호 설정" 섹션
시작 번호 유효 범위 정의 필요: "1 이상, 설정된 자릿수의 최대값(예: 5자리 → 99999) 이하의 정수만 입력 가능" + 초과 입력 시 에러 메시지 정의.
🔵 편집 오류 (Info) — 오탈자, 문법 오류, 포맷 깨짐

#11일련번호 자릿수 표시문구 — 문장 편집 미완성

편집 오류
"선택한예를 들어, 이 값을 5로 설정할 경우 숫자 앞에 0이 붙어 00001, 00002와 같은 5자리 숫자로 생성됩니다."

"선택한"으로 시작하는 문장이 중간에 잘리고, 바로 이어 "예를 들어"가 붙어 비문이 되었습니다. 이전 편집 시 문구를 수정하다가 앞부분 삭제가 누락된 것으로 보입니다.

📌 출처: "일련번호 자릿수 설정" > 표시문구
"예를 들어, 이 값을 5로 설정할 경우…" 또는 "선택한 자릿수에 따라…"로 정리.

#12만료 경고 팝업 문구 — 대괄호 중복 및 줄바꿈 누락

편집 오류
"… [[New Case ID_2] 순으로 계속해서 부여됩니다.번호 중복이나 …"

① [[New Case ID_2]에서 대괄호가 [[로 중복되어 있고, ② 문장 사이에 줄바꿈/공백이 빠져 "부여됩니다.번호"로 이어지며, ③ 전체 팝업 문구가 한 줄로 이어져 가독성이 떨어집니다. 팝업 내용이 개발자에게 직접 전달되는 문구이므로 정확한 포맷 필요.

📌 출처: "팝업 노출 조건" > 팝업 내용
대괄호 수정: [New Case ID_2], 문장 간 줄바꿈 추가, 전체 문구를 블록 형태로 재정리.
