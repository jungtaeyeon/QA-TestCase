문서 A Case ID.md › 변경 이력 (26.04.17)
"번호 조합에서 Year+Month의 조합 삭제"
문서 B Case ID.md › Year (연도 표시 및 리셋 정책) — 툴팁 문구
"연도 선택 시 매년 초, 연월 선택 시 매월 초에 일련번호가 '00001'로 초기화됩니다. N/A 선택 시(Chip을 추가하지 않았을 경우) 초기화 없이 번호가 계속 누적됩니다."
26.04.17 기획 변경에서 "Year+Month(연월)" 조합이 삭제되어 Year 설정 테이블에는 2-digit year / 4-digit year / N/A 세 가지만 남음. 그러나 툴팁 문구에는 삭제된 "연월 선택 시 매월 초에 초기화" 표현이 그대로 잔류하여 기능 존재 여부에 대한 혼동을 야기함.
권장 조치: ① 月 단위 초기화(Year+Month) 기능이 실제 존재하는지 기획자 확인 필요. ② 기능이 삭제 확정이라면 툴팁 문구에서 "연월 선택 시 매월 초에 일련번호가 '00001'로 초기화됩니다." 부분 삭제 요청. ③ 확인 전까지 Year 리셋 관련 TC 작성 보류.
CONFLICT-002 🔴 충돌 영향도: Medium
Case Level 식별자 예시 내 자릿수 불일치 — 3자리(001) vs 5자리(00002) 혼용
문서 A Case ID.md › ICSR 식별자 관리 사양서 — Case vs Report Identifier 노출 사양 테이블
Case Level 표시 형식 예: 202604ST001, 202604ST00002
동일 셀 내 두 예시에서 일련번호 자릿수가 3자리(001)와 5자리(00002)로 불일치. Zero-padding 규칙에 따르면 설정된 자릿수가 고정 적용되어야 하므로, 두 예시가 서로 다른 설정 시나리오를 의도한 것인지 명확히 기재 필요.
권장 조치: 동일 자릿수 기준으로 예시 통일 또는 "3자리 설정 예: 001 / 5자리 설정 예: 00001" 형식으로 구분 명시 요청.

"자유 텍스트" / "Free Text" / "Custom Text" 세 표현 혼용
표현 A Case ID.md › 섹션 헤더 / Chip 이름
섹션명: "자유 텍스트 추가" / Chip명: "TXT=자유 텍스트"
표현 B, C Case ID.md › 자유 텍스트 추가 — 본문
"Free Text를 체크박스 형태로 선택 가능" / "선택 시 [Custom Text] 블록 제공"
동일 기능을 "자유 텍스트", "Free Text", "Custom Text" 세 가지로 혼용. 화면 UI 레이블·버튼명이 어떤 표현을 따라야 할지 불명확하며, TC 작성 시 기준 용어가 없음.
권장 조치: 화면 표시 레이블 기준 표현 하나로 통일 후 기획서 전체에 일관 적용 요청. (제안: 섹션명·Chip명을 기준으로 "자유 텍스트" 또는 영문 "Custom Text"로 통일)
EXPR-002 🟠 표현 불일치 영향도: Medium
"조직 식별자" vs "조직명" 혼용
표현 A Case ID.md › 개요 및 정책
"추후 국가코드, 조직명을 포함하여 C.1.1 R3 필드에 매핑"
표현 B Case ID.md › 기본 구조 및 국가 코드 로직 / ICSR 식별자 관리 사양
"[국가코드]-[조직 식별자]-[설정된 번호 조합]" / "조직 식별자 입력 — 최소 1글자 최대 10글자"
C.1.1 식별자 구조의 두 번째 구성 요소가 개요에서는 "조직명", 이후 전 섹션에서는 "조직 식별자"로 표기됨. 같은 필드를 가리키는지 또는 다른 개념인지 불명확.
권장 조치: "조직 식별자"가 기술적으로 더 정확 (사용자 입력 임의 텍스트이므로 "명칭"보다 "식별자"가 적합). 개요 섹션 "조직명" → "조직 식별자"로 수정 요청.

"보고 구분" vs "Type of Report" 혼용

근거 Case ID.md — 조직 식별자 / 자유 텍스트 / 시작 번호 Validation 섹션들
조직 식별자 "최소 1글자 최대 10글자" / 자유 텍스트 "최대 6글자" / 시작 번호 "최후 번호보다 커야 함", "[최대값-400] 초과 불가" — 각 위반 시 에러 메시지 없음

권장 조치: 각 Validation 규칙에 대응하는 에러 메시지 문구 기획서에 추가 요청. 메시지 문구 없는 항목은 참고용 TC 작성 가능하나 메시지 검증 TC는 확정 후 작성.

MISSING-004 🟡 누락 영향도: Medium
자동 자릿수 확장 발생 시 사용자 알림 방식 미정의
근거 Case ID.md › Case ID 자릿수 자동 확장
"서비스 중단 없이 시스템이 자동으로 자릿수를 확장하여 번호를 부여" — 알림 방식 미정의
자동 확장은 서비스 중단 없이 처리된다고 명시되어 있으나, 확장 발생 시 Admin에게 알림을 보내는지(팝업/이메일/Dashboard 알림), 알림이 없는지에 대한 정의가 없음.

MISSING-005 🟡 누락 영향도: Medium
PRD Chip 포함 설정 + 케이스 내 제품 없을 때 동작 불명확 ("활성화됨" vs "UNK 표시" 모순)
근거 Case ID.md › Product Abbreviation 추가
"노출 조건: 케이스 생성 시 '제품 불러오기'를 통해 제품 정보가 케이스 내에 포함된 경우에만 해당 값이 활성화됨. 값 자체가 없을 경우 UNK으로 표시"
"활성화됨"(제품 있을 때만 노출)과 "UNK으로 표시"(값 없을 때 대체값)가 의미상 충돌. PRD Chip이 Case ID 포맷에 포함된 상태에서 케이스에 제품 정보가 없는 경우 — PRD 부분이 Case ID에서 생략되는지(비활성화), UNK으로 대체 포함되는지 불명확.
권장 조치: "제품 없을 때 PRD Chip 동작" 명확화 요청. 예: "제품 정보 있으면 약어, 없으면 UNK으로 항상 Case ID에 포함됨" 또는 "제품 없으면 PRD 부분 자체가 Case ID에서 생략됨" 중 정의.
🔵 확인 필요 사항 (Confirmation) — 5건
CONFIRM-001 🔵 확인 필요 영향도: High ⛔ TC 작성 보류
Import 옵션 B 선택 시 중복 C.1.1 처리 방법 미정의
관련 기획 Case ID.md › Import 설정 — 옵션 B
"B. C.1.1, C.1.8.1 유지 — C.1.1 & C.1.8.1: 파일 내 데이터를 모두 그대로 사용"
옵션 B로 Import 시 파일 내 C.1.1 값을 그대로 사용하는데, 해당 값이 이미 시스템에 존재하는 케이스와 중복될 경우의 처리 방법(에러 반환, 덮어쓰기, 경고 후 진행 등)이 정의되어 있지 않음. E2B(R3) 규칙상 C.1.1 중복은 식별 오류를 야기할 수 있음.
권장 조치: 옵션 B 사용 시 C.1.1 중복 발생 시 처리 정책 정의 필요. 확인 전까지 Import 옵션 B TC 작성 보류.
CONFIRM-002 🔵 확인 필요 영향도: Medium
자릿수 중간 변경 시 기존 케이스의 Tracker 표시 방식 미정의
관련 기획 Case ID.md › 일련번호 자릿수 설정
"중간에 케이스 자릿수 변경 시, 이후 생성되는 케이스부터 적용"
변경 이후 케이스는 새 자릿수 적용이 명확. 그러나 변경 전 생성된 케이스(구 자릿수)가 Tracker에서 어떻게 표시되는지, 정렬/검색 시 혼재된 자릿수 처리 방식이 미정의.
권장 조치: 자릿수 변경 전 케이스의 Tracker 표시 방식(기존 형식 유지 vs 새 형식 마이그레이션) 확인 요청.
CONFIRM-003 🔵 확인 필요 영향도: Medium ⛔ TC 작성 보류
[Do not show again] 설정 후 새 자릿수 체계 전환 시 팝업 재노출 여부
관련 기획 A Case ID.md › 사용자 인터랙션 및 제어
"[Do not show again] 체크 후 [Confirm] 클릭: 해당 팝업을 더 이상 노출하지 않음"
관련 기획 B Case ID.md › 사용자 인터랙션 및 제어
"자동 미노출: 일련번호가 완전히 초과되어 새로운 자릿수 체계(1XXXXX...)로 넘어간 경우에는 기존 팝업 노출을 중단"

권장 조치: 자릿수 체계 전환 후 "Do not show again" 설정 리셋 여부 및 새 팝업 노출 조건 정의 요청. 확인 전까지 관련 TC 보류.
CONFIRM-004 🔵 확인 필요 영향도: Low
"다음 로그인 시 재노출" — 세션 만료/재접속 포함 여부 불명확
